# Newsi（知新）MVP PRD

## 文档信息

- 产品名称：Newsi（知新）
- 版本：MVP v1.0
- 状态：可进入规划阶段
- 日期：2026-03-21
- 平台：Web

## 1. 产品概述

### 1.1 一句话定义

Newsi 是一个面向个人知识工作者的 Web 产品。用户用自然语言写下自己想长期关注的领域，系统每天自动围绕这些兴趣生成一份站内日报，而不是依赖大众热点、推荐算法或通用新闻流。

### 1.2 背景

在 AI 时代，信息问题已经不再是“获取不到”，而是“信息过载、被推荐机制扭曲、以及大众热点与个人真实关注点不匹配”。

知识工作者通常只关心一组相对狭窄但持续变化的主题，例如 AI Agent、设计工具、开发平台、创业生态或某些特定行业。现有产品大多优化的是停留时长和大众流行度，而 Newsi 要优化的是“由用户自己定义的信息相关性”。

### 1.3 核心价值主张

Newsi 让用户主动定义“什么值得关注”，再通过 AI 的每日工作流总结这些领域里当天发生了什么，把被动刷信息流的行为，替换成一种安静、克制、个人化的日报阅读体验。

## 2. 目标用户

### 2.1 核心用户群体

面向个人知识工作者，包括但不限于：

- 产品经理
- 设计师
- 开发者
- 研究者
- 投资人
- 独立开发者
- 创业者

### 2.2 用户特征

- 为工作、学习或决策持续跟踪信息
- 更在意信息信噪比，而不是信息量
- 不想每天自己手动搜索
- 对通用新闻 App 和社交信息流不满意
- 比起嘈杂的 feed，更偏好安静、专注的阅读体验

### 2.3 核心用户问题

用户并不缺“新闻”，他们缺的是一个稳定答案：

“今天，在我关心的领域里，发生了什么变化？”

## 3. 产品目标与非目标

### 3.1 MVP 目标

MVP 要验证的核心假设是：

> 只要系统能够基于用户自己的兴趣描述，持续生成一份足够相关的站内日报，用户就愿意把长期的信息追踪任务交给 AI。

具体目标包括：

- 让用户通过一段自然语言定义自己的长期关注范围
- 每天自动在产品内生成一篇日报
- 提供安静、可连续阅读的当日阅读体验
- 保留历史日报，便于回看
- 保持产品极简、可理解、可形成习惯

### 3.2 非目标

MVP 不包含以下内容：

- 邮件发送
- Push 通知
- 社交分享
- 多人协作
- 每个用户多套兴趣配置
- 用户手动确认 AI 拆解结果
- 自定义来源管理
- 自建抓取系统或 RSS 接入
- 趋势图表、分析看板或洞察仪表盘
- 历史日报搜索
- PDF 或 Markdown 导出

## 4. 产品原则

- 用户定义的相关性优先于平台定义的热门度
- 阅读体验优先于信息密度
- 每日综合整理优先于无限信息流
- 极简配置优先于细粒度控制
- AI 辅助理解优先于原始结果堆叠

## 5. MVP 范围摘要

MVP 只包含 3 个用户可见页面：

- `Today`：阅读今天的日报
- `Archive`：按日期浏览历史日报
- `Topics`：创建或编辑用户的兴趣描述

系统行为为：

- 用户输入一段自然语言兴趣描述
- 系统将其保存为用户的长期关注说明
- 系统每天调用一次具备实时网络查询能力的大模型
- 系统生成当天日报
- 用户在站内阅读结果

## 6. 用户故事

### 6.1 核心用户故事

- 作为一名知识工作者，我希望直接用自然语言写下我关注的内容，而不是配置复杂规则。
- 作为用户，我希望系统每天自动准备好一份日报，而不是我自己每天去搜索。
- 作为阅读者，我希望今天的日报是经过整理和筛选的，而不是零散信息拼接。
- 作为回访用户，我希望能查看历史日报，从而形成长期观察。
- 作为用户，我希望在关注重点变化时，随时修改自己的兴趣描述。

### 6.2 边缘场景故事

- 作为新用户，如果我保存兴趣后当天日报还没生成，我希望知道接下来会发生什么。
- 作为用户，如果系统正在生成或生成失败，我希望看到清晰状态，而不是空白页。
- 作为用户，我希望修改兴趣描述后，只影响未来日报，不覆盖过去的历史内容。

## 7. 信息架构

### 7.1 导航结构

左侧固定导航，包含 3 个入口：

- Today
- Archive
- Topics

### 7.2 页面职责

#### Today

核心消费页面。用于展示当天日报，强调安静、长文式、沉浸式阅读。

#### Archive

历史日报列表页面，用于按日期回看过去生成的日报。

#### Topics

唯一配置页面，用于输入和修改用户的自然语言兴趣描述。

## 8. 核心用户流程

### 8.1 首次使用

1. 用户进入产品。
2. 用户进入 `Topics` 页面。
3. 用户写下一段自然语言，描述自己长期关注的方向。
4. 用户保存这段描述。
5. 系统将其保存为当前生效的兴趣档案。
6. 如果当日日报尚未存在，产品需要告诉用户第一份日报会在何时出现。

### 8.2 日常使用

1. 系统每天在固定时间启动一次日报生成任务。
2. 系统读取用户当前保存的兴趣描述。
3. 系统调用具备联网能力的大模型，请它基于该描述整理当天的综合摘要。
4. 系统将生成结果写入当天日报。
5. 用户打开 `Today` 页面进行阅读。

MVP 的日报调度规则：

- 日报所属日期以用户账户保存的时区为准。
- 首次使用时，系统应从浏览器推断用户时区，并在服务端保存。
- MVP 中该账户时区不提供用户手动修改能力。
- `Today` 页面在该时区的 00:00 切换到新的一天。
- 自动日报任务应在该时区的每天 07:00 触发一次。
- 如果用户在当天调度时间之后才首次保存兴趣描述，则第一份有保障的日报从下一自然日开始生成。

### 8.3 持续管理

1. 用户在 `Topics` 页面编辑兴趣描述。
2. 新描述从下一次生成开始生效。
3. 已经存在的历史日报不被回写或覆盖。

## 9. 功能需求

### 9.1 兴趣档案

#### 描述

每个用户只有一份当前生效的兴趣档案，以自然语言文本表达。

#### 需求

- 系统必须允许用户创建兴趣描述。
- 系统必须允许用户编辑并覆盖现有兴趣描述。
- 系统必须持久化保存最新版本。
- 系统必须在未来日报生成时使用最新保存版本。

#### 不在范围内

- 多个兴趣档案
- 向用户暴露版本历史
- 关键词权重
- 包含词 / 排除词
- 来源偏好
- topic 级别的结构化配置 UI

### 9.2 每日日报生成

#### 描述

系统基于用户保存的兴趣描述，并调用具备实时网络研究能力的大模型，每天自动生成一篇日报。

#### 需求

- 系统必须每天在固定时间自动运行一次生成任务。
- 系统每个用户每个自然日最多只生成一份正式日报。
- 系统必须将生成状态与最终内容分开存储。
- 系统必须在失败后自动重试。
- 系统不得要求用户手动触发每日生成。
- 自然日定义和调度时间必须基于用户账户保存的时区。

#### 假设

- 外部模型具备实时网页研究或基于搜索结果完成综合整理的能力。
- MVP 阶段产品本身不拥有独立检索层。
- 系统可以在首次会话中自动推断并持久化用户时区，而无需在 MVP 中提供专门设置项。

#### 不在范围内

- 用户手动重跑
- 同一天多版本日报
- 用户可见的来源配置

### 9.3 Today 页面

#### 描述

Today 是产品最核心的阅读页面，用于展示当天日报。

#### 必需状态

- 尚未配置兴趣描述
- 已配置兴趣描述，但当日日报尚未生成
- 正在生成
- 生成失败
- 生成成功

#### 需求

- 当用户尚未配置兴趣描述时，页面必须引导其前往 `Topics`。
- 当当日日报尚不可用时，页面必须说明日报会在固定时间自动生成。
- 当系统正在生成时，页面必须展示明确的处理中状态。
- 当生成失败时，页面必须展示明确失败状态，并说明系统会自动重试。
- 当生成成功时，页面必须完整展示日报内容。

### 9.4 Archive 页面

#### 描述

Archive 用于按日期展示历史日报，供用户后续回看。

#### 需求

- 系统必须按时间顺序展示历史日报列表。
- 每条列表项至少包含日期和日报标题。
- 每条列表项应包含预计阅读时长或相近元信息。
- 用户点击后必须能打开对应日报详情。

#### 不在范围内

- 搜索
- 过滤
- 按主题聚类
- 跨日期趋势对比

### 9.5 Topics 页面

#### 描述

Topics 是用户唯一的配置页面，用于填写和修改兴趣描述。

#### 需求

- 页面必须展示当前兴趣描述（如已存在）。
- 页面必须提供文本输入区域。
- 页面必须提供保存动作。
- 空状态必须说明：这里输入的内容会驱动未来的每日日报。

## 10. 日报内容规格

### 10.1 内容目标

日报应回答以下问题：

- 今天，在用户关心的领域里，发生了什么变化？
- 为什么这些变化值得注意？
- 哪些信号最值得记住？

### 10.2 内容结构

建议每日日报使用以下结构：

#### 页头

- 日期
- 日报标题，例如 `Today's Synthesis`
- 一句简短导语，概括今天的重点

#### 正文

按 3 到 5 个主题段落组织。每个段落包括：

- 一个主题标题
- 2 到 4 段综合性摘要
- 一组关键要点
- 可选的一句“为什么值得关注”

#### 页尾

- 简短结束语或结束标记

### 10.3 内容规则

- 优先做总结，而不是穷举罗列
- 优先保证相关性，而不是覆盖面
- 聚焦变化、信号和发展
- 避免生成“通用新闻流”风格文本
- 避免夸张和耸动表达
- 目标阅读时长控制在 5 到 8 分钟

### 10.4 来源

MVP 采取“摘要优先”原则，来源链接不是硬性要求。

这意味着：

- 如果模型稳定返回来源信息，产品可以展示，但 UI 和验收标准不依赖它。
- 产品在 MVP 阶段不应把自己表述为严格可验证的引用工具。

## 11. 系统状态与失败处理

### 11.1 用户可见状态

Today 页面至少应支持以下状态：

- `unconfigured`
- `scheduled`
- `generating`
- `failed`
- `ready`

### 11.2 失败处理原则

- 失败不能导致空白页。
- 不论当天是否失败，历史日报都必须可访问。
- 面向用户的文案应简单、非技术化。
- 重试应在后台自动发生。

### 11.3 重试策略

- 首次生成失败后，系统应自动重试，最多 3 次。
- 重试应尽可能在同一个日报日内完成。
- 如果所有重试都失败，则当天日报保持 `failed` 状态，不应悄悄挪到其他日期补写。
- 只有在仍属于同一个日报日期的前提下，后续成功的重试才可以更新当日失败记录。

## 12. 数据模型

### 12.0 账户模型

MVP 默认采用登录用户模型。Newsi 为每个用户账户保存一份当前兴趣档案及对应的日报历史。

### 12.1 User

用于将数据绑定到单个用户的最小账户实体。

可能字段包括：

- `user_id`
- `account_timezone`

### 12.2 InterestProfile

可能字段包括：

- `user_id`
- `interest_text`
- `updated_at`

### 12.3 DailyDigest

可能字段包括：

- `user_id`
- `digest_date`
- `status`
- `title`
- `intro`
- `content`
- `reading_time`
- `created_at`
- `updated_at`

## 13. 设计与体验方向

### 13.1 产品气质

整体体验应当是安静的、带编辑感的、克制而有意图的。它不应看起来像社交 feed、指标堆叠型 dashboard 或交易终端。

### 13.2 体验原则

- 极简导航
- 强调排版与留白
- 空状态文案克制清晰
- 阅读优先布局
- 低交互密度

### 13.3 与原型关系

你提供的原型已经明确了 MVP 的正确方向：

- 左侧栏导航
- 大面积阅读主画布
- 克制稀疏的 Archive 列表
- 简洁的兴趣输入页
- 强调安静、专注、综合整理的整体气质

## 14. 成功指标

### 14.1 激活指标

- 兴趣描述保存率
- 首篇日报生成成功率
- 首篇日报打开率

### 14.2 留存指标

- 7 日回访率
- 一周内打开多篇日报的用户占比
- Archive 回看率

### 14.3 产品质量信号

- 兴趣描述修改率
- 日报生成失败率
- 日报阅读完成度代理指标，例如停留时长或滚动深度

## 15. 风险与取舍

### 15.1 黑盒风险

由于用户不查看 topic 拆解结果，系统在结果偏离时会显得不透明。

应对方式：

- 保持兴趣编辑足够简单
- 明确告知日报是基于当前保存的兴趣描述生成
- 不要假装产品的控制精度高于实际能力

### 15.2 外部模型质量风险

MVP 依赖外部大模型完成实时研究与综合整理，覆盖度、时效性和摘要质量都可能波动。

应对方式：

- 把产品定位为“每日日报综合工具”，而不是“完整情报平台”
- 保持 MVP 范围足够窄
- 只有当质量成为核心阻碍时，再考虑补建检索层

### 15.3 习惯形成风险

如果日报内容每天都显得重复、浅层或缺少变化，用户会停止回访。

应对方式：

- 强调“今天的变化”
- 减少反复解释长期背景
- 保持段落短小且以信号为导向

## 16. 验收标准

当以下条件全部满足时，可认为 MVP 完成：

- 用户可以创建并保存一段自然语言兴趣描述。
- 系统会将该描述保存为未来生成任务的生效输入。
- 系统每天自动生成一篇站内日报。
- Today 页面可正确展示 `unconfigured`、`scheduled`、`generating`、`failed`、`ready` 等状态。
- 生成成功后的日报可在 Today 页面正常阅读。
- 历史日报可在 Archive 中查看。
- 用户可以在 Topics 页面编辑兴趣描述。
- 编辑兴趣描述只影响未来日报，不覆盖历史日报。

## 17. 延后到 MVP 之后的问题

以下问题明确延后处理，不应阻塞 MVP 规划：

- 后续是否支持多套 topic / profile？
- 来源链接是否要成为一等 UI 对象？
- 用户是否可以手动重跑当天日报？
- 产品是否要支持通知或邮件？
- Archive 是否需要搜索和过滤？

## 18. 最终范围说明

Newsi MVP 不是一个完整新闻平台，也不是通用知识管理系统。它是一个最小但清晰的 Web 产品：用户只需要定义一次自己关心什么，系统就把这份长期关注说明转化为一份每天可阅读、可回看的站内 AI 日报。
