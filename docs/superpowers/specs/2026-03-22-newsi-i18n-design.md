# Newsi 多语言支持设计

- 日期：2026-03-22
- 状态：Approved for planning
- 关联项目：Newsi MVP
- 目标范围：在不改动现有 URL 结构的前提下，为产品增加英文/中文界面文案切换能力

## 1. 背景

当前 Newsi 面向用户的界面文案基本为硬编码英文，包括：

- 根布局中的 `<html lang="en">`
- `Signin` 页的说明与按钮
- `Today`、`Archive`、`Topics` 中的空状态与状态提示
- 左侧导航与表单按钮

本次需求要求：

- 默认语言为英文
- 用户可在界面中手动切换到中文
- 语言偏好仅保存在当前浏览器
- 不改动现有路由路径
- 只翻译界面文案，不翻译用户输入内容和 AI 生成内容

## 2. 目标

本次改动需要实现：

- 新用户首次访问时默认看到英文界面
- 用户可在应用内切换中英文
- 切换结果在当前浏览器中持久化
- 刷新页面或跳转页面后，界面保持所选语言
- 服务端首屏输出即为正确语言，避免客户端 hydration 后再替换文案

## 3. 非目标

本次不做：

- 路由级国际化，不新增 `/en/...` 或 `/zh/...`
- 根据浏览器语言自动覆盖用户选择
- 将语言偏好写入数据库或绑定账号
- 翻译用户输入的 `Topics` 内容
- 翻译 `DigestView` 中的 AI 标题、摘要、正文、要点
- 引入重量级国际化框架
- 支持两种以上语言

## 4. 产品决策

### 4.1 语言策略

- 支持语言：`en`、`zh`
- 默认语言：`en`
- URL 保持不变，例如 `/today`、`/archive`、`/topics`
- 语言切换入口放在应用共享界面中，用户显式触发切换

### 4.2 持久化策略

- 使用浏览器 cookie 保存语言偏好
- cookie 缺失、为空或非法时，回退为 `en`
- 不与认证态耦合，预览模式和正式登录态使用同一套 locale 解析逻辑

### 4.3 翻译边界

只翻译系统界面文案，包括：

- 导航名称
- 页面标题与描述
- 表单 placeholder、按钮、说明文案
- 空状态与状态面板文案
- 系统兜底标题，例如归档中的 `Digest scheduled`

保持原样的内容包括：

- 用户输入的兴趣描述
- 数据库中已存储的 digest 标题和正文
- AI provider 输出的 `DigestView` 内容

## 5. 方案选择

本次采用轻量自建 i18n 方案，而不是引入第三方国际化框架。

选择理由：

- 当前项目规模较小，文案数量有限
- 需求明确要求不改 URL，和许多基于路由 locale 的方案不完全匹配
- 大部分页面是服务端渲染，更适合在服务端直接解析 locale 并输出目标文案
- 未来如果需要升级为账号级语言偏好，仍可复用本次的消息字典和翻译接口

## 6. 架构设计

### 6.1 Locale 核心模块

新增一个轻量 locale 模块，负责：

- 定义受支持语言：`en`、`zh`
- 定义默认语言：`en`
- 定义 locale cookie key
- 提供 locale 校验与回退逻辑
- 提供读取当前请求语言的统一入口

建议接口职责：

- `isSupportedLocale(value)`
- `normalizeLocale(value)`
- `getRequestLocale()`

### 6.2 消息字典

新增本地消息字典文件，按功能域分组管理，例如：

- `layout`
- `signin`
- `today`
- `archive`
- `topics`
- `states`

设计要求：

- `en` 作为基准字典
- `zh` 结构必须与 `en` 对齐
- 页面或组件通过稳定 key 获取文案，而不是在业务代码中散落条件分支

### 6.3 翻译读取方式

采用“页面层读取翻译，组件层尽量只接收字符串”的原则：

- 根布局读取 locale，并设置 `<html lang>`
- 页面级 Server Component 根据 locale 取出文案
- 基础展示组件继续保持无业务状态，接收已经翻译好的 props

这样可以减少共享组件对全局上下文的依赖，也更便于测试。

## 7. 页面与组件改造

### 7.1 根布局

`src/app/layout.tsx` 需要：

- 读取 locale cookie
- 将 `<html lang>` 从固定 `en` 改为动态值
- 使用当前语言输出默认 metadata 文案

### 7.2 共享布局

`AppShell` / `SideNav` 需要：

- 展示导航项的双语文案
- 提供一个轻量语言切换控件
- 切换动作写入 cookie 后触发当前页面刷新

语言切换控件要求：

- 可见性稳定，不依赖登录态
- 当前语言高亮明确
- 不引入额外复杂交互

### 7.3 Signin

需要翻译：

- 品牌辅助说明
- 未配置认证时的提示
- 预览入口按钮文案

### 7.4 Topics

需要翻译：

- 页面标题
- 引导描述
- textarea 的 `aria-label`
- placeholder
- `Save interests`
- `Clear interests`

保留不变：

- `interestText` 表单值本身
- 浏览器时区写入逻辑

### 7.5 Today

需要翻译：

- 无 Topics 时的 empty state
- `scheduled` / `generating` / `failed` / `unavailable` 的 label 与 body
- 预览模式下对应的状态文案

保留不变：

- digest 数据读取逻辑
- `DigestView` 的标题、intro、sections 内容

### 7.6 Archive

需要翻译：

- 无归档时的 empty state
- 持久化未配置时的提示
- 归档默认标题：
  - `Digest scheduled`
  - `Digest generating`
  - `Digest failed`
  - `Untitled digest`

保留不变：

- 数据库中已有 digest 的真实标题
- 归档列表的数据结构与跳转逻辑

## 8. 数据流设计

### 8.1 首次访问

1. 请求进入服务端
2. 根布局读取 locale cookie
3. cookie 不存在，回退到 `en`
4. 页面以英文完成首屏渲染

### 8.2 手动切换语言

1. 用户点击共享布局中的语言切换控件
2. 系统写入 locale cookie，例如 `zh`
3. 当前页面刷新或重定向回原页面
4. 服务端重新读取 cookie
5. 新语言文案在首屏输出

### 8.3 后续访问

- 同一浏览器后续访问继续读取已保存的 locale cookie
- 认证前后不重置语言偏好
- 预览模式与正式环境均遵循相同规则

## 9. 错误处理与回退策略

- locale cookie 缺失时回退到 `en`
- locale cookie 非法时回退到 `en`
- 中文字典缺失某个 key 时，运行时回退到英文文案
- 语言切换失败时，不影响主业务流程，仅保持当前语言
- 多语言逻辑不能阻断 `Signin`、`Today`、`Archive`、`Topics` 的现有业务路径

## 10. 测试策略

### 10.1 单元测试

新增或更新测试覆盖：

- locale 解析函数的合法值、非法值、默认值回退
- 字典结构一致性检查
- 切换动作对 cookie 写入行为的验证

### 10.2 集成测试

更新现有页面测试并补充语言场景：

- 默认无 cookie 时渲染英文文案
- `locale=zh` 时 `Signin` 显示中文界面文案
- `locale=zh` 时 `Topics` 显示中文标题、placeholder、按钮
- `locale=zh` 时 `Today` 的 empty state 和状态面板显示中文
- `locale=zh` 时 `Archive` 的 empty state 和默认归档标题显示中文

### 10.3 E2E 测试

可增加一条最小路径验证：

1. 打开应用
2. 切换到中文
3. 刷新页面
4. 确认界面仍保持中文
5. 切回英文并再次确认

该项为建议项，不作为本次第一阶段上线阻塞条件。

## 11. 实施边界

本次改动应控制在“界面文案国际化”范围内，不应顺带扩散为：

- digest 内容语言化
- 用户偏好设置中心
- 时区、日期格式的全面本地化
- 数据库存量内容迁移
- 路由与 SEO 国际化体系

## 12. 验收标准

满足以下条件即可认为本次设计达标：

- 默认访问时界面为英文
- 用户能在界面中手动切换到中文
- 语言偏好在当前浏览器内可持久化
- 页面刷新后语言保持不变
- 系统主要界面文案在中英文之间可正确切换
- digest 真实内容保持原样，不被错误翻译
- 非法 locale 输入不会导致页面报错
